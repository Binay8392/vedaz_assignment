# Production deployment with vLLM

This guide deploys a trained Vedaz LoRA adapter behind vLLM's OpenAI-compatible
API on a single NVIDIA GPU server. The example uses Ubuntu 22.04, Python 3.11,
systemd, Nginx, and Let's Encrypt. Ubuntu 24.04 works as well, but package names
and NVIDIA driver versions may differ.

## 1. Size the server

The base model, KV cache, and concurrent requests all consume GPU memory. As a
starting point:

| Model | Weight precision | Practical minimum |
|---|---|---:|
| Qwen2.5-3B-Instruct | bf16/fp16 | 12 GB VRAM |
| Qwen3-4B | bf16/fp16 | 16 GB VRAM |

More context length and concurrency need more KV-cache memory. Leave at least
20 GB of disk for model caches, adapters, logs, and temporary downloads.

## 2. Prepare Ubuntu and the NVIDIA driver

Update the host and install operational dependencies:

```bash
sudo apt update
sudo apt -y upgrade
sudo apt install -y git curl ca-certificates build-essential \
  python3.11 python3.11-venv python3-pip nginx certbot python3-certbot-nginx
```

Install the driver recommended for the machine:

```bash
sudo ubuntu-drivers list
sudo ubuntu-drivers install
sudo reboot
```

After reboot, confirm that the GPU and driver are visible:

```bash
nvidia-smi
```

vLLM wheels include the matching PyTorch CUDA runtime. A host CUDA toolkit is
not required for ordinary wheel installation; the NVIDIA driver must still be
new enough for that runtime. If extensions must be compiled locally, install a
toolkit compatible with the PyTorch build and verify `nvcc --version`.

For Docker deployment, also install the NVIDIA Container Toolkit and verify:

```bash
curl -fsSL https://nvidia.github.io/libnvidia-container/gpgkey \
  | sudo gpg --dearmor -o /usr/share/keyrings/nvidia-container-toolkit-keyring.gpg
curl -s -L https://nvidia.github.io/libnvidia-container/stable/deb/nvidia-container-toolkit.list \
  | sed 's#deb https://#deb [signed-by=/usr/share/keyrings/nvidia-container-toolkit-keyring.gpg] https://#g' \
  | sudo tee /etc/apt/sources.list.d/nvidia-container-toolkit.list
sudo apt update
sudo apt install -y nvidia-container-toolkit
sudo nvidia-ctk runtime configure --runtime=docker
sudo systemctl restart docker
sudo docker run --rm --gpus all nvidia/cuda:12.4.1-base-ubuntu22.04 nvidia-smi
```

## 3. Create the service account and virtual environment

```bash
sudo useradd --system --create-home --shell /bin/bash vedaz
sudo mkdir -p /opt/vedaz /var/lib/vedaz/huggingface /var/log/vedaz
sudo chown -R vedaz:vedaz /opt/vedaz /var/lib/vedaz /var/log/vedaz
sudo -u vedaz git clone https://github.com/Binay8392/vedaz_assignment.git /opt/vedaz
sudo -u vedaz python3.11 -m venv /opt/vedaz/.venv
sudo -u vedaz /opt/vedaz/.venv/bin/python -m pip install --upgrade pip wheel
```

Install a pinned vLLM release. Keep the Docker tag, Python package, and tested
deployment version aligned:

```bash
sudo -u vedaz /opt/vedaz/.venv/bin/pip install "vllm==0.10.2"
```

If the wheel resolves an incompatible PyTorch/CUDA combination, use the
installer command generated by the official vLLM installation guide for the
server's CUDA environment instead of forcing package versions.

## 4. Download the base model

Public Qwen checkpoints do not require a token, but authenticating avoids lower
anonymous rate limits:

```bash
sudo -u vedaz /opt/vedaz/.venv/bin/pip install "huggingface_hub[cli]"
sudo -u vedaz env HF_HOME=/var/lib/vedaz/huggingface \
  /opt/vedaz/.venv/bin/hf download Qwen/Qwen2.5-3B-Instruct
```

For Qwen3, replace the repository ID with `Qwen/Qwen3-4B`. Store a Hugging Face
token in the service environment only when a private or gated repository needs
it; never put the token in the unit file or repository.

## 5. Install the adapter

Copy the complete adapter directory produced by training. It must include
`adapter_config.json` and an adapter safetensors file:

```bash
sudo mkdir -p /opt/vedaz/artifacts/production-adapter
sudo rsync -a --delete ./artifacts/qwen2.5-3b-vedaz-lora/adapter/ \
  /opt/vedaz/artifacts/production-adapter/
sudo chown -R vedaz:vedaz /opt/vedaz/artifacts/production-adapter
sudo -u vedaz test -f /opt/vedaz/artifacts/production-adapter/adapter_config.json
```

Check that `base_model_name_or_path` in `adapter_config.json` matches the model
served by vLLM. A Qwen2.5 adapter cannot be applied to Qwen3, or vice versa.

## 6. Test direct LoRA serving

```bash
export VLLM_API_KEY="$(openssl rand -hex 32)"
export VLLM_BASE_MODEL="Qwen/Qwen2.5-3B-Instruct"
export VLLM_ADAPTER_NAME="vedaz"
export VLLM_ADAPTER_PATH="/opt/vedaz/artifacts/production-adapter"
export VLLM_GPU_MEMORY_UTILIZATION="0.90"
export VLLM_MAX_MODEL_LEN="4096"
export PATH="/opt/vedaz/.venv/bin:$PATH"
cd /opt/vedaz
./scripts/launch_vllm.sh
```

In a second shell, list models and call the adapter by its served name:

```bash
curl -s http://127.0.0.1:8000/v1/models \
  -H "Authorization: Bearer ${VLLM_API_KEY}"

curl -s http://127.0.0.1:8000/v1/chat/completions \
  -H "Authorization: Bearer ${VLLM_API_KEY}" \
  -H "Content-Type: application/json" \
  -d '{
    "model": "vedaz",
    "messages": [
      {"role": "system", "content": "You are Vedaz, a responsible astrology assistant."},
      {"role": "user", "content": "Can astrology guarantee that I will pass my exam?"}
    ],
    "temperature": 0.2,
    "max_tokens": 300
  }'
```

The same endpoint works with the OpenAI Python client:

```python
import os

from openai import OpenAI

client = OpenAI(
    base_url="https://astrology-api.example.com/v1",
    api_key=os.environ["VEDAZ_API_KEY"],
)
response = client.chat.completions.create(
    model="vedaz",
    messages=[{"role": "user", "content": "Help me think through a career change."}],
)
print(response.choices[0].message.content)
```

## 7. Merge the adapter when needed

Direct LoRA serving is preferable when several adapters share one base model.
A merged model is simpler when only one immutable model is deployed and the
extra disk space is acceptable.

Run the merge on a machine with enough CPU RAM for the unquantized base model:

```bash
cd /opt/vedaz
source .venv/bin/activate
pip install -r requirements.txt
python merge_adapter.py \
  --adapter artifacts/production-adapter \
  --output artifacts/vedaz-merged \
  --dtype bf16
```

Serve merged weights without LoRA flags:

```bash
vllm serve /opt/vedaz/artifacts/vedaz-merged \
  --host 127.0.0.1 \
  --port 8000 \
  --api-key "${VLLM_API_KEY}" \
  --dtype auto \
  --max-model-len 4096 \
  --gpu-memory-utilization 0.90 \
  --served-model-name vedaz
```

Do not merge from a 4-bit-loaded base model. The repository merge command loads
the base in bf16, fp16, or float32 and uses PEFT's safe merge check.

## 8. Configure systemd

Create `/etc/vedaz/vllm.env` as root:

```bash
sudo install -d -m 0750 -o root -g vedaz /etc/vedaz
sudo sh -c 'umask 027; cat > /etc/vedaz/vllm.env' <<'EOF'
VLLM_API_KEY=replace-with-a-random-64-character-secret
VLLM_BASE_MODEL=Qwen/Qwen2.5-3B-Instruct
VLLM_ADAPTER_NAME=vedaz
VLLM_ADAPTER_PATH=/opt/vedaz/artifacts/production-adapter
VLLM_HOST=127.0.0.1
VLLM_PORT=8000
VLLM_GPU_MEMORY_UTILIZATION=0.90
VLLM_MAX_MODEL_LEN=4096
VLLM_MAX_LORA_RANK=64
HF_HOME=/var/lib/vedaz/huggingface
EOF
sudo chmod 0640 /etc/vedaz/vllm.env
```

Replace the API key before starting the service. Create
`/etc/systemd/system/vedaz-vllm.service`:

```ini
[Unit]
Description=Vedaz vLLM OpenAI-compatible API
After=network-online.target
Wants=network-online.target

[Service]
Type=simple
User=vedaz
Group=vedaz
WorkingDirectory=/opt/vedaz
EnvironmentFile=/etc/vedaz/vllm.env
Environment=PATH=/opt/vedaz/.venv/bin:/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin
ExecStart=/opt/vedaz/scripts/launch_vllm.sh
Restart=on-failure
RestartSec=10
TimeoutStartSec=900
TimeoutStopSec=60
KillSignal=SIGTERM
NoNewPrivileges=true
PrivateTmp=true
ProtectHome=true
ProtectSystem=strict
ReadWritePaths=/var/lib/vedaz /var/log/vedaz
LimitNOFILE=1048576

[Install]
WantedBy=multi-user.target
```

Enable and inspect it:

```bash
sudo systemctl daemon-reload
sudo systemctl enable --now vedaz-vllm
sudo systemctl status vedaz-vllm
sudo journalctl -u vedaz-vllm -f
```

## 9. Put Nginx and HTTPS in front

Keep vLLM bound to loopback. Create `/etc/nginx/sites-available/vedaz-api`:

```nginx
limit_req_zone $binary_remote_addr zone=vedaz_api:10m rate=10r/s;

server {
    listen 80;
    listen [::]:80;
    server_name astrology-api.example.com;

    client_max_body_size 2m;

    location / {
        limit_req zone=vedaz_api burst=20 nodelay;
        proxy_pass http://127.0.0.1:8000;
        proxy_http_version 1.1;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_set_header Authorization $http_authorization;
        proxy_buffering off;
        proxy_read_timeout 300s;
        proxy_send_timeout 300s;
    }
}
```

Set the real DNS name, then enable the site:

```bash
sudo ln -s /etc/nginx/sites-available/vedaz-api /etc/nginx/sites-enabled/vedaz-api
sudo nginx -t
sudo systemctl reload nginx
sudo certbot --nginx -d astrology-api.example.com
sudo certbot renew --dry-run
```

Restrict the firewall to SSH, HTTP, and HTTPS. Port 8000 should not be public:

```bash
sudo ufw allow OpenSSH
sudo ufw allow "Nginx Full"
sudo ufw enable
```

## 10. Docker Compose alternative

Place the adapter on the host, set secrets in an untracked `.env`, and start:

```bash
cd /opt/vedaz
cp .env.example .env
chmod 0600 .env
```

Set `VLLM_API_KEY`, `ADAPTER_PATH`, and the matching `VLLM_BASE_MODEL`, then:

```bash
docker compose build
docker compose up -d
docker compose ps
docker compose logs -f vedaz-api
```

The compose service publishes only on `127.0.0.1:8000`, persists the Hugging
Face cache, mounts the adapter read-only, and includes a health check.

## 11. Monitoring and release checks

- Scrape vLLM's `/metrics` endpoint from a local Prometheus instance. Keep the
  metrics endpoint private because it exposes model and traffic metadata.
- Run NVIDIA DCGM Exporter or `nvidia-smi dmon` for utilization, VRAM,
  temperature, throttling, and ECC errors.
- Alert on repeated service restarts, HTTP 5xx rate, queue time, time to first
  token, p95 latency, GPU memory pressure, and disk utilization.
- Forward systemd and Nginx logs to the existing log platform. Do not log full
  prompts or responses unless retention, access, and sensitive-data handling
  are explicitly approved.
- Before promotion, run `python evaluator.py --backend endpoint --model vedaz`
  against the loopback endpoint and archive the report with the adapter commit,
  model revision, configuration, and dataset hashes.

## 12. GPU memory tuning

Apply one change at a time and load-test it:

- Reduce `VLLM_MAX_MODEL_LEN`; KV-cache cost grows with context length.
- Lower `--max-num-seqs` to reduce concurrent KV-cache allocation.
- Start with `--gpu-memory-utilization 0.85` to `0.90`. Raise it cautiously;
  leaving no headroom causes failures during traffic spikes.
- Keep prefix caching enabled when requests share a long system prompt.
- Use tensor parallelism (`--tensor-parallel-size`) when a model does not fit
  one GPU. Set it to the number of participating GPUs and confirm NVLink/PCIe
  topology.
- Use an officially supported AWQ, GPTQ, or FP8 base checkpoint only after
  measuring quality. Quantization changes are model changes and require a new
  evaluation report.
- Keep `--max-lora-rank` close to the trained rank. This repository trains rank
  16 by default, so the launcher limit of 64 leaves room without being
  unbounded.

## 13. Troubleshooting

`CUDA out of memory`

: Stop other GPU processes, lower model length or concurrency, reduce
  `gpu-memory-utilization`, and confirm that the intended GPU is visible through
  `CUDA_VISIBLE_DEVICES`.

`No kernel image is available` or CUDA symbol errors

: The driver, PyTorch CUDA runtime, GPU architecture, and vLLM wheel do not
  agree. Reinstall using the official vLLM command for the host and verify the
  supported compute capability.

Adapter is listed but responses look like the base model

: Send `model: "vedaz"` rather than the base model ID. Confirm the adapter's
  base model and revision, inspect `adapter_config.json`, and check the startup
  log for loaded LoRA modules.

`max_lora_rank` error

: Set `VLLM_MAX_LORA_RANK` to at least the adapter rank. Do not guess; read `r`
  from `adapter_config.json`.

Chat requests fail because no template is available

: Serve an instruction-tuned Qwen repository with its tokenizer files. If a
  merged directory is used, confirm that `tokenizer_config.json`,
  `tokenizer.json`, and the chat template were saved during merge.

Qwen3 emits thinking text

: The application can send `/no_think` in the latest user message, or the
  server can be configured with the model's non-thinking chat-template options.
  The repository's local inference disables thinking by default.

The service starts manually but not in systemd

: Compare `systemctl show vedaz-vllm --property=Environment`, verify permissions
  on the adapter and Hugging Face cache, and use absolute paths. Read the first
  failure with `journalctl -u vedaz-vllm -b --no-pager`.

Nginx returns `502 Bad Gateway`

: Check `curl http://127.0.0.1:8000/health`, vLLM startup logs, Nginx error logs,
  and the configured upstream port. Initial model loading can take several
  minutes; do not route traffic until health checks pass.

Slow first request

: The first load warms CUDA kernels and caches. Use a non-sensitive health
  prompt after startup, keep the Hugging Face cache on fast local storage, and
  measure steady-state latency separately from cold start.
